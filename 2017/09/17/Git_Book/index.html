Initialized empty Git repository in .git/

Git doesn’t care whether you start with a completely empty directory or with a directory full of files. In either case, the process of converting the directory into a Git repository is the same.

Git will create .git at the top of your project. Whereas CVS and Sub-version place the revision information in CVS and .svn subdirectories with in each of your project’s directories.

Adding a File to your repository

git init creates a new Git repository. Initially, each Git repository is empty. You can explicitly deposit it in the repository.

Now, Git knows that the file added is to remain in the repository. However, so far Git has merely staged the file, an interim step before committal. Git separates the add and commit to avoid volatility.

It would be disruptive, confusing and time-consuming if it owuld be to update each time you add, remove, or change a file.

Running git status reveals current state.

In addition to actual changes to the directory and to file contents, Git records several other pieces of metadata with each commit, including a log message and the author of the change.

You can also use your favorite editor to edit the comment.

After you commit the addition of the new file into the repository, git status indicates that there are no outstanding, staged changes to be committed.

Configuring the Commit author

You can also specify the name and email using GIT_AUTHOR_NAME and GIT_AUTHOR_EMAIL environment variables.

Last: Configuration files

  • .git/config: Repository-specific configuration settings manipulated with the –file option or by default. These settings have the highest precedence.
  • ~/.gitconfig: User-specific configuration settings manipulated with the –global option.
  • /etc/gitconfig: System-wide configuration settings manipulated with the –system option. These system settings have the lowest precedence. (Sometimes in /usr/local/etc/gitconfig)
